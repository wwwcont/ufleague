# Product + UX Foundation Specification
## Мобильное веб-приложение футбольного турнира (Frontend-first, MVP foundation)

## 0) Рамки и допущения

### 0.1 Текущий этап
- Проект создается с нуля.
- Фокус реализации: **frontend**.
- Backend на текущем этапе не реализуется.
- На первом этапе используются **моковые данные** (fixtures / in-memory / local JSON).
- Приложение должно запускаться локально через `npm`.

### 0.2 Целевые платформы
- Приоритет устройства: **мобильные телефоны**.
- Поддержка desktop обязательна, но через **mobile-first подход**.
- Базовый таргет viewport для проектирования: `360x800` / `390x844`.

### 0.3 Продуктовая цель foundation-этапа
Сформировать единый каркас продукта и UX-правила, чтобы:
1. начать быструю frontend-разработку экранами P0,
2. обеспечить консистентный пользовательский опыт,
3. заранее подготовить интеграционные точки для backend,
4. не перепроектировать навигацию и shell после MVP.

- **Уровень 2 (внутренние):** Match Details, Team Details, Player Details, Bracket.
- **Уровень 3 (аккаунт):** Login, Profile.

## 2) Обязательный UI-каркас (App Shell)

## 2.1 Shell-структура
На большинстве public-страниц используется единый shell:
1. **Fixed Header** (фиксированная верхняя шапка)
2. **Scrollable Content Area** (контентная прокручиваемая область)
3. **Fixed Bottom Navigation** (фиксированное нижнее меню)

## 2.2 Инварианты shell
- Header и Bottom Nav не демонтируются при переходах между табами и основными разделами.
- Меняется только контентная область.
- Пользователь всегда видит, что находится внутри одного приложения.

## 2.3 Исключения
- Login может использовать упрощенный shell (без bottom nav), если это улучшает фокус на действии.
- Profile после входа может использовать стандартный shell, если включается в основную навигацию.

## 2.4 Безопасные отступы и зоны
- Учитываются safe-area отступы (`env(safe-area-inset-top/bottom)`).
- Минимальная интерактивная зона элементов: **44x44 px**.
- Bottom Nav должен иметь достаточный нижний padding для устройств с gesture-bar.

## 3) UX-принципы и правила поведения

## 3.1 Mobile-first UX
- Сначала проектируются мобильные сценарии (портретная ориентация).
- На mobile приоритет у вертикального скролла и коротких блоков.
- Desktop — адаптация через расширение сетки, а не новый UX с нуля.

## 3.2 Thumb-friendly navigation
- Основные действия — в нижней зоне экрана (bottom nav, primary actions).
- Ключевые CTA и фильтры не должны требовать постоянного reach в верхнюю часть.
- Свайпы допускаются только как ускорители, не как единственный способ навигации.

## 3.3 Predictable flows
- Списки ведут в деталку по универсальному паттерну “list → detail → related entities”.
- В каждой деталке есть явные переходы в связанные сущности (матч → команда → игрок).
- Один и тот же тип данных имеет одинаковое визуальное представление во всех разделах.

## 3.4 Persistent shell behavior
- Header: стабильный контейнер контекста страницы.
- Bottom Nav: постоянный глобальный ориентир.
- Back-навигация в header для внутренних страниц (кроме Home).

## 3.5 Page hierarchy
- Home = обзор + быстрые входы.
- Разделы списков (Matches/Teams/Players) = browse и фильтрация.
- Детальные страницы = deep dive + переходы в связанные сущности.

## 3.6 Content priority
1. Актуальные и ближайшие матчи.
2. Турнирное положение (таблица / стадия).
3. Сущности команды и игроки.
4. Учетная запись.

## 3.7 Section rhythm (ритм секций)
Для мобильного экрана:
- Hero / headline блок (опционально)
- Контролы (фильтры/переключатели)
- Основной список/контент
- Secondary links / related content

Рекомендуемый вертикальный ритм:
- Отступ между большими секциями: 16–24 px.
- Отступ внутри карточек: 12–16 px.

## 3.8 Когда использовать hero blocks
Использовать hero только когда нужно быстро задать контекст страницы:
- Home: да (следующий ключевой матч / текущий тур).
- Match Details: да (счет, статус, тайминг, команды).
- Team/Player Details: ограниченно, как identity block.
- Списки (Matches/Teams/Players): обычно без hero, чтобы не “съедать” экран.

## 3.9 Когда использовать фильтры
Фильтры обязательны там, где есть длинные списки и альтернативные срезы:
- Matches: дата, тур, статус (upcoming/live/finished).
- Teams: группа/дивизион (если применимо), сортировка.
- Players: команда, позиция.
- Standings: группа/лига, если структура турнира это требует.

Правило mobile: первичный фильтр — чипы/сегменты в 1 строку + “More filters” в bottom sheet.

## 3.10 Поведение поиска
- Поиск глобальный по сущностям: команды, игроки, матчи.
- Выдача сгруппирована по типам сущностей.
- Должны быть состояния: initial, typing, results, empty, error.
- Debounce запроса (для будущего API) и мгновенная локальная фильтрация на моках.
- Из результата всегда прямой переход в деталку сущности.

## 3.11 Навигация list ↔ details
- Клик по карточке в списке ведет в соответствующую деталку.
- В деталке есть блок “Связанные сущности”:
  - Match → Team, Player, Standings.
  - Team → Players, Matches, Standings.
  - Player → Team, Matches.
- Back возвращает пользователя в предыдущий контекст (с сохранением скролла и фильтра).

## 3.12 Турнирная сетка на mobile
- Горизонтально скроллируемый канвас раундов.
- Каждый раунд — отдельная вертикальная колонка.
- Карточка матча компактная, но с обязательными: команды, счет/статус, стадия.
- Есть быстрый прыжок к текущему раунду (“Current round”).
- Поддерживается pinch/zoom опционально; если нет — fallback на step-zoom кнопки.

## 4) Экранная карта и спецификация экранов

## 4.1 Home
**Цель:** быстро дать обзор турнира и вход в ключевые сценарии.

**MVP приоритет:** P0.

**Главные блоки:**
- Hero: ближайший / центральный матч.
- Блок “Сегодня/Ближайшие матчи”.
- Мини-таблица (top positions) с CTA в Standings.
- Быстрые входы: Matches, Teams, Bracket, Search.

**Задачи пользователя:**
- Понять, что происходит в турнире сейчас.
- Перейти в нужный раздел за 1–2 тапа.

**Визуальный приоритет:** ближайшие матчи и CTA переходов.

**Mobile поведение:**
- Компактные карточки, один столбец.
- Ключевые блоки выше fold.

**Desktop адаптация:**
- Двухколоночный layout (overview + secondary widgets).

## 4.2 Matches
**Цель:** найти нужный матч по времени/туру/статусу.

**MVP приоритет:** P0.

**Главные блоки:**
- Переключатель периода (дата / тур).
- Фильтры статуса (upcoming/live/finished).
- Список матчей (карточки).

**Задачи пользователя:**
- Просмотреть расписание и результаты.
- Открыть Match Details.

**Визуальный приоритет:** список матчей и статус матча.

**Mobile поведение:**
- Sticky row фильтров под header.
- Быстрая вертикальная прокрутка списка.

**Desktop адаптация:**
- Доп. колонка с расширенными фильтрами.

---

## 4.3 Match Details
**Цель:** дать полную карточку матча и связи с другими сущностями.

**MVP приоритет:** P0.

**Главные блоки:**
- Hero scoreboard (команды, счет, статус, дата/время).
- Таймлайн/события (минимум: голы, карточки, статусные события).
- Составы/игроки (минимально — ключевые игроки).
- Related links: Team A, Team B, Standings.

**Задачи пользователя:**
- Проверить результат/статус.
- Перейти к команде/игроку.

**Визуальный приоритет:** scoreboard и статус матча.

**Mobile поведение:**
- Hero фиксирован в верхней части контента, остальное секционно.
- Крупные тач-таргеты для переходов к командам.

**Desktop адаптация:**
- 2 колонки (основной контент + сайд-блоки).

**MVP приоритет:** P0.

**Главные блоки:**
- Поисковая строка внутри раздела.
- Фильтры/сортировки (по группе/алфавиту/форме).

- Найти команду.

**Mobile поведение:**

**Desktop адаптация:**
- Расширенная сетка карточек.

## 6) Особые UX-требования

## 6.1 Турнирная сетка на мобильных
- Основной паттерн: горизонтальный скролл по раундам.
- Каждая колонка раунда подписана и визуально отделена.
- Карточки матчей единообразны по высоте для читаемости связей.
- В верхней зоне: mini-map индикатор текущего раунда.
- Тап по карточке открывает Match Details.

## 6.2 Постоянное нижнее меню
- Состав пунктов (минимум): Home, Matches, Teams, Standings, Search.
- Активный пункт явно подсвечен.
- Иконка + короткий лейбл обязательны.
- Меню фиксировано и не скрывается при обычном скролле.
- На внутренних деталках нижнее меню сохраняется для глобального перехода.

## 6.3 Back navigation
- На всех внутренних страницах (Details, Bracket, Search results-context, Login, Profile) есть кнопка Back в header.
- На Home кнопки Back нет.
- При Back восстанавливаются: предыдущий scroll position, выбранные фильтры, вкладка.
- Если страница открыта прямым deep-link без истории, Back ведет в логический родительский раздел:
  - Match Details → Matches
  - Team Details → Teams
  - Player Details → Players

## 6.4 Поведение шапки (Header)
- Header всегда фиксирован сверху.
- Слева: Back (когда применимо) или логотип/название.
- Центр: заголовок текущего экрана.
- Справа: контекстное действие (например Search shortcut на релевантных экранах).
- Высота стабильная, без “прыжков” между разделами.

## 6.5 Переходы между матчем, командой, игроком и таблицей
Кросс-навигация обязательна и двусторонняя:
- Из Match Details:
  - в Team Details обеих команд,
  - в Player Details ключевых игроков,
  - в Standings (позиции команд).
- Из Team Details:
  - в Match Details (календарь/результаты команды),
  - в Player Details (состав),
  - в Standings.
- Из Player Details:
  - в Team Details,
  - в Match Details последних матчей.
- Из Standings:
  - в Team Details по тапу на команду,
  - оттуда дальше в матчи/игроков.

Принцип: пользователь должен пройти путь **Match ↔ Team ↔ Player ↔ Standings** без тупиков.

## 7) UX state model (для frontend readiness под будущий backend)

Для каждого экрана P0 должны быть заранее заложены состояния:
- `loading`
- `refreshing`
- `success`
- `empty`
- `error`

Требования:
- Skeleton или placeholder для загрузки списков.
- Понятные empty-state тексты с CTA “вернуться/сбросить фильтры”.
- Retry в error-состояниях.
- Контракты данных формализованы через frontend модели, даже при моках.

## 8) Routing foundation

## 8.1 Предлагаемая карта маршрутов
- `/` → Home
- `/matches` → Matches
- `/matches/:matchId` → Match Details
- `/teams` → Teams
- `/teams/:teamId` → Team Details
- `/players` → Players
- `/players/:playerId` → Player Details
- `/standings` → Standings
- `/bracket` → Bracket
- `/search` → Search
- `/login` → Login
- `/profile` → Profile

## 8.2 Правила маршрутизации
- Детальные страницы всегда имеют стабильный id-параметр.
- Поддерживаются deep-links.
- При прямом входе в деталку приложение отображает логический parent в back-flow.

## 9) Минимальные нефункциональные UX-требования

- Базовая производительность mobile: первая meaningful отрисовка ключевого контента без визуальных лагов.
- Доступность: контрастность, читаемые размеры текста, фокус-состояния (для desktop клавиатуры).
- Консистентность UI-компонентов между разделами.
- Подготовка к i18n: тексты и лейблы не “зашиваются” в компоненты напрямую.

## 11) Definition of Done для Foundation-этапа

Foundation считается готовым, если:
1. Утверждена карта экранов и приоритеты P0/P1/P2.
2. Зафиксированы shell-инварианты и back-навигация.
3. Для всех P0 экранов описаны цель, блоки, действия и адаптация mobile/desktop.
4. Зафиксированы кросс-переходы между Match/Team/Player/Standings.
5. Определена mobile-стратегия для Bracket.
6. Есть базовая routing-структура и state model под будущий backend.
